## Входы

Стартуй от:
- согласованных входных документов
- текущего запроса пользователя
- текущего YAML, если это режим `edit`

## Модель

Рабочая иерархия:

- `modules`
- внутри каждого модуля: `steps`
- внутри каждого шага: `links`

## Семантика

- YAML должен выражать логику программы, а не комментарии о проекте.
- Модуль отвечает на вопрос, зачем этот кусок существует в системе.
- Шаг отвечает на вопрос, что происходит внутри модуля.
- `purpose` модуля не должен дублировать детальную логику, уже написанную в шагах.

## Жёсткие Правила

- Не выдумывай логику, которая не обсуждалась.
- Если при authoring не хватает критичной информации — остановись и спроси, а не заполняй дыры догадками.
- Если ты не можешь честно собрать читаемый сквозной модульный маршрут, остановись и задай вопросы.
- Известные ветки должны быть структурными путями, а не текстом, спрятанным внутри описания шага.
- `input` и `output` остаются опциональными. Заполняй их только там, где они действительно понятны.
- Этот skill пишет YAML. Он не выполняет reviewer-оценку.

## Маршрут, Порядок И Связи

- Порядок модулей в массиве `modules` выражает основной маршрут системы.
- Порядок шагов в массиве `steps` — это путь по умолчанию внутри модуля. Явные `links` переопределяют этот порядок для ветвлений и нелинейной маршрутизации.
- Первый шаг в массиве `steps` модуля является его точкой входа. Кросс-модульные ссылки должны указывать на него.
- Поле `to` в `links` может ссылаться на шаг в другом модуле — это основной механизм кросс-модульных связей.
- Условия на links должны оставаться короткими.
- Идентификаторы модулей и шагов должны отражать их смысловую роль, а не порядок в маршруте. Префикс модуля в ID шага не обязателен, поскольку шаг уже находится в контексте своего модуля.

## Выбор Режима

### Create

Используй `create`, когда ещё нет удовлетворительного YAML или пользователь хочет получить первый рабочий документ. Шаги ниже — рекомендуемый порядок работы, а не жёсткий чеклист.

1. Сначала раздели систему на модули.
   - Предлагай нарезку смело — не жди, пока пользователь явно определит каждый модуль.
   - Смещайся в сторону разделения **разных логик**, а не разных **фаз одной логики**. Фазы одного исполнителя в одном сквозном flow — это шаги внутри модуля, а не отдельные модули.
   - Самый сильный сигнал новой границы модуля: смена исполнителя, режима обработки, внешнего сервиса или зоны ответственности.
   - Не превращай мелкие проверки и локальные edge cases в отдельные модули.
   - Антипаттерн: модуль из 1-2 шагов, у которого тот же исполнитель что у соседнего модуля. Это сигнал что модули нужно объединить.
   - Проверяй нарезку вопросом: «у этих двух модулей один и тот же актор/исполнитель и один сквозной flow?» Если да — это один модуль.

2. Построй полный маршрут по модулям.
   - Маршрут системы должен читаться целиком на уровне модулей.
   - У каждого известного модуля должно быть видимое место в этом маршруте.
   - Если модуль ещё недоопределён, всё равно оставь его в маршруте, а не выноси за пределы документа.

3. Детализируй только зрелые части.
   - Раскладывай модуль на steps только тогда, когда его внутренняя логика действительно понятна.
   - Внутри одного модуля можно смешивать детальные шаги и один оставшийся грубый блок, если это помогает не выдумывать лишнюю структуру.
   - Главный сигнал нового шага: смена действия или состояния процесса.

4. Честно представляй неизвестное.
   - Если известно, что альтернативная ветка существует, но она ещё не проработана, создай для неё грубый блок вместо того, чтобы скрывать или выдумывать детали.
   - Если модуль ещё сырой, оставь один внутренний логический блок или шаг с полным Markdown-описанием.
   - Если есть две правдоподобные модульные нарезки и одна явно сильнее, выбирай её и продолжай. Если неоднозначность остаётся рискованной, спрашивай.

5. Запиши результат в новый YAML-файл.
   - Создай файл, а не просто выведи YAML в чат.
   - Добавляй дополнительный текст только тогда, когда нужно отметить блокеры или задать уточняющие вопросы.

### Edit

Используй `edit`, когда уже существует текущий YAML-документ.

1. Считай текущий YAML базовым документом.
2. Вноси только ту логику, которая поддержана согласованными входами и текущим запросом.
3. При необходимости ты можешь существенно перестроить структуру. 
4. Не выдумывай новые маршруты, условия или внутренние детали, которые не были согласованы.
5. Если запрошенное изменение затрагивает неясную связанную логику, остановись и спроси, а не протаскивай допущения внутрь YAML.
6. Верни обновлённый YAML-документ.

## Опциональный Handoff В Review

Если другой workflow просит подготовить контекст для отдельного reviewer-агента:
- для первичного `create`: передай новый YAML плюс короткое описание логики, из которой он был собран
- для `edit`: передай старую и новую программную логику в прозе; предполагается, что YAML diff будет приложен внешней логикой workflow

## Формат Выхода

- Предпочитай YAML как основной ответ.
- Держи прозу короткой и операционной.
- Не добавляй вердикт ревью.
- Не объясняй каждую строку YAML, если пользователь об этом не попросил.

## Рабочая Форма

Рабочая форма по умолчанию, когда в проекте ещё нет более сильной локальной схемы:

```yaml
modules:
  - id: semantic-module-id
    purpose: Короткая формулировка, зачем этот модуль существует.
    input: Опционально, только когда уже ясно.
    output: Опционально, только когда уже ясно.
    depends_on:                    # Опционально — зависимости от других модулей
      - other-module-id
    steps:
      - id: semantic-step-id
        body_markdown: |
          Основная логика шага.
        links:
          - to: another-step-id  # может быть шаг в этом или другом модуле
            condition: Опциональное короткое условие.
```

`depends_on` — список id модулей, от которых зависит данный модуль. Используй, если модуль предполагает что другой уже отработал или подготовил окружение, но между ними нет прямых step-to-step links. Визуализируется как пунктирная линия зависимости между рамками модулей.

Форма для недоопределённого модуля:

```yaml
modules:
  - id: coarse-module
    purpose: Короткая формулировка, зачем этот модуль существует.
    steps:
      - id: coarse-module-overview
        body_markdown: |
          Один логический блок, который честно объясняет модуль,
          не выдумывая преждевременную внутреннюю структуру.
```

Канонический пример смешанной зрелости — в `references/example.md`. Прочитай его перед первым `create`.

## Рекомендации По Стилю body_markdown

Содержимое `body_markdown` рендерится как Markdown внутри визуального узла на canvas. Ширина узла подстраивается под контент. Эти рекомендации помогают сделать результат читаемым:

- Используй списки (`-`) для перечислений вместо сплошного текста.
- Используй inline code (`` ` ``) для технических терминов: названий полей, переменных, API endpoints, статусов.
- Используй блоки кода (` ``` `) для команд, конфигов, технических фрагментов. Блоки кода рендерятся мельче и моноширинно.
- Разбивай длинный текст на абзацы через пустую строку. Один абзац — одна мысль.
- Каждый шаг описывает одно действие или одно состояние. Если описание разрастается в стену текста, это сигнал что шаг стоит разбить на несколько.
- Для грубого логического блока (недоопределённый модуль) длинный текст допустим.
- Не добавляй заголовки (`#`, `##`) внутри `body_markdown`.
- Думай о визуальном балансе. Короткое действие — короткое описание. Команда — в блоке кода. Не раздувай простые шаги и не сжимай сложные.
