# Deep Research

**Ты — координатор.** Твоя задача: составить brief, обсудить, запустить, следить за прогрессом. Субагент получает промпт и инструменты автоматически.

## Фаза 0: Согласование (ОБЯЗАТЕЛЬНА)

**Никогда не запускать субагента без подтверждения пользователя.**

### Шаг 1: Собрать контекст
- Прочитай `/root/another-openclaw/research/INDEX.md` — сводки всех прошлых ресёрчей
- Поискать в `/root/another-openclaw/` — всё что связано с темой
- Проверить experience — `/root/another-openclaw/experience/` — опыт по релевантным сайтам/темам
- Вспомнить что обсуждали

### Шаг 2: Составить brief
```
📋 **Ресёрч: [тема — 1 строка]**

**Что уже знаем:**
- [факт 1]
- [факт 2]

**Что выяснить:**
1. [вопрос → конкретный deliverable]
2. [вопрос → конкретный deliverable]

**Жёсткие требования (обязательны):**
- [требование 1 — конкретное, измеримое, проверяемое]
- [требование 2]

**Ожидаемый результат:** [таблица/список/сравнение/инструкция]
```

Предварительный ответ на вопрос — только если есть обоснованное предположение. Не угадывай.

### Шаг 3: Обсуждение
Между brief и запуском может быть целый разговор. Пользователь уточняет, добавляет вопросы, корректирует. Показать модель и thinking level. Ждать явного подтверждения: «ок», «давай», «запускай».

Если тема не привязана к одной стране — указать в task что поиск нужно вести по нескольким странам. Researcher сам определит какие страны релевантны.

### Шаг 4: Выбор модели
По умолчанию: `gpt-5.4`, thinking: `xhigh`.
Пользователь может изменить при обсуждении.

## Фаза 1: Запуск

### Определить тип

**Новый** — тема впервые.
**Продолжение** — есть предыдущий ресёрч, нужно углубить/расширить (см. секцию «Продолжение ресёрча»).

### 1. Создать файлы

```
/root/another-openclaw/research/<slug>/research.md    ← шаблон с brief
/root/another-openclaw/research/<slug>/progress.md    ← пустой шаблон
```

**research.md шаблон:**
```markdown
# Research: [Тема]
Дата: YYYY-MM-DD

## Brief
[Вопросы из brief]

## Rolling Summary
(обновляется по ходу исследования)

## Key Findings
(нумерованный список с ссылками [S#])

## Sources
(нумерованный список URL)

## Open Questions
[Начальные вопросы]

## Log
- HH:MM — research запущен
```

**progress.md шаблон:**
```markdown
# Progress: [Тема]
```

### 2. Обновить INDEX.md

Дописать запись в `/root/another-openclaw/research/INDEX.md` со статусом «в процессе».

### 3. Запустить субагента

```
sessions_spawn(
  task: "<задание — см. ниже>",
  agentId: "researcher",
  model: "gpt-5.4",
  thinking: "xhigh",
  label: "research-<slug>",
  cleanup: "keep",
  runTimeoutSeconds: 7200
)
```

### Формат task

**Обычный ресёрч:**
```
Проведи глубокое исследование. Работай МИНИМУМ [N] ЧАСА. Не торопись.

## Задание
**Тема:** [тема]
**Что нужно выяснить:** [brief]
**Минимальное время работы:** [N часов]
**Файл результатов:** /root/another-openclaw/research/<slug>/research.md
**Файл прогресса:** /root/another-openclaw/research/<slug>/progress.md
**Папка опыта:** /root/another-openclaw/experience/

Начинай с понимания задания (Фаза 0), потом работай.
```

Минимальное время обсуждается с пользователем при согласовании brief. Если пользователь не указал — спросить.

**С BROWSER_FIRST:**
Добавить секцию: `## Режим extraction: BROWSER_FIRST`

**С дополнительным контекстом:**
Добавить: `## Предоставленный контекст` с собранными данными.

**Follow-up (новый spawn):**
Добавить: `## Follow-up` с путём к базовому исследованию и инструкциями [BASE-S#].

## Фаза 2: Мониторинг

Субагент пишет в progress.md:
- **PLAN** — план исследования
- **CLARIFICATION** — вопросы (субагент **ЖДЁТ** ответа)
- **PROGRESS** — каждые 3-5 находок
- **QUESTION** — вопросы (субагент НЕ останавливается)
- **FINAL** — готово

### Обработка CLARIFICATION (блокирующая)

Субагент ждёт. Показать пользователю, получить ответ.
Дописать в progress.md: `[HH:MM] COMMAND ANSWER: <ответ>`

### Команды координатора

Дописать в progress.md:

| Команда | Формат |
|---|---|
| Остановить | `[HH:MM] COMMAND STOP` |
| Сменить направление | `[HH:MM] COMMAND REFOCUS: <направление>` |
| Ответить | `[HH:MM] COMMAND ANSWER: <ответ>` |
| Добавить вопрос | `[HH:MM] COMMAND ADD_QUESTION: <вопрос>` |

## Фаза 3: Завершение

Когда FINAL или announce:
1. Прочитать research.md
2. Дать пользователю краткое резюме
3. Предложить прочитать полный файл или задать вопросы
4. **ОБЯЗАТЕЛЬНО** обновить INDEX.md:

```
## <slug>
- **Дата:** YYYY-MM-DD
- **Статус:** завершён
- **Сводка:** <2-3 предложения из Rolling Summary>
- **Путь:** research/<slug>/
- **Session:** <sessionKey из announce>
```

**⚠️ БЕЗ ЗАПИСИ SESSION В INDEX.MD ПРОДОЛЖЕНИЕ РЕСЁРЧА НЕВОЗМОЖНО.**
SessionKey приходит в announce после завершения субагента — найди его и запиши.

## Продолжение ресёрча

Если пользователь хочет продолжить существующий ресёрч — **всегда продолжать в том же research.md**, не создавать новый файл.

### Через sessions_send (основной способ)

Сессии субагентов живут долго (setup ставит `archiveAfterMinutes` примерно на 1 год).

1. Найти в INDEX.md запись → взять sessionKey из поля `Session:`
2. Обсудить с пользователем что нужно доработать
3. `sessions_send(sessionKey, "<что доработать>")` — субагент просыпается с полным контекстом и продолжает работу в том же research.md
4. Обновить INDEX.md: статус «в процессе»

### Через новый spawn (fallback)

Если sessionKey нет или сессия пропала:

1. Новый `sessions_spawn` с follow-up task — указать путь к существующему research.md
2. Субагент перечитает research.md, увидит что уже найдено, и продолжит работу не дублируя поиски
3. Обновить INDEX.md: новый sessionKey, статус «в процессе»

## Follow-up сценарии

| Сценарий | Brief |
|---|---|
| Углубить | «В базе X поверхностно. Углуби: числа, механизмы, примеры» |
| Закрыть вопросы | «Остались вопросы: [список]. Найди ответы» |
| Крашнутый | «Воркер не завершился. Продолжи с того места» |
| Другой ракурс | «База про X с точки A. Исследуй с точки B» |

## Ограничения

- `runTimeoutSeconds: 7200` (макс 2 часа)
